Jito Bundles: The Complete Technical Guide

Everything you need to know about Jito bundle submission — from basic mechanics to advanced tip optimization and failure recovery.

April 5, 2026 10 min read Technical Guide

Jito bundles are the primary mechanism for ordered transaction submission on Solana. If you're building MEV bots, liquidation systems, or any application that needs guaranteed execution order, Jito is the tool. This guide covers everything from basic concepts to production optimization.

What Are Jito Bundles?

A Jito bundle is a set of 1-5 transactions that are submitted to the Jito block engine for atomic, ordered inclusion in a Solana block. The block engine forwards bundles to validators running the Jito-Solana client, which processes them with two guarantees:

  1. Ordering: Transactions in the bundle execute in the exact sequence you specify
  2. Atomicity: Either all transactions in the bundle land, or none do (if any fails, the entire bundle is dropped)

This is fundamentally different from standard Solana transaction submission, where there are no ordering guarantees and transactions are processed independently.

Bundle Structure

A bundle consists of:

  • 1-5 transactions — serialized Solana transactions, each independently signed
  • A tip transaction — the last transaction in the bundle, transferring SOL to a Jito tip account as payment for inclusion

The tip transaction is what incentivizes validators to process your bundle over competing bundles. Higher tips get priority. The tip goes to one of the 8 Jito tip distribution accounts (rotated per epoch).

// Basic bundle: strategy TX + tip TX
let strategy_tx = create_arbitrage_transaction(&pool_a, &pool_b, amount)?;
let tip_tx = create_tip_transaction(
    &payer,
    &jito_tip_account,   // Current epoch's tip account
    tip_amount_lamports,  // Dynamic tip
)?;

let bundle = vec![strategy_tx, tip_tx];
jito_client.send_bundle(&bundle).await?;

Tip Account Rotation

Jito rotates tip accounts across epochs. The 8 tip accounts are publicly known, and your bot should resolve the current epoch's active tip account before each submission. Sending tips to a wrong-epoch account still works but may be processed with lower priority.

Tip Optimization Strategies

The tip amount is the most critical parameter in bundle submission. Too low and your bundle doesn't land. Too high and you eat into your profit margin. Here's how to optimize:

Dynamic Tip Calculation

Don't use static tips. Calculate them dynamically based on:

  • Expected profit: Never tip more than 50-70% of expected profit. A minimum profit threshold ensures you don't execute losing trades.
  • Recent landing rates: Track your landing rate over the last 100 bundles. If it drops below your target (e.g., 40%), increase tips. If it exceeds target, decrease to save margin.
  • Competition signals: For highly competitive opportunities (major pair arbs), tip more aggressively. For long-tail opportunities, conservative tips work.
fn calculate_tip(expected_profit: u64, recent_landing_rate: f64) -> u64 {
    let base_ratio = if recent_landing_rate < 0.3 {
        0.70  // Aggressive: landing rate too low
    } else if recent_landing_rate < 0.5 {
        0.50  // Normal
    } else {
        0.35  // Conservative: landing rate is good
    };

    let tip = (expected_profit as f64 * base_ratio) as u64;
    let min_tip = 10_000;  // 0.00001 SOL minimum

    tip.max(min_tip)
}

Per-Slot-Leader Analysis

Not all validators process Jito bundles the same way. Some have lower minimum tip thresholds, some are more responsive. Track which validators land your bundles and adjust tips per-leader:

  • Maintain a HashMap<Pubkey, TipStats> per validator
  • When a slot starts, look up the leader and use their historical tip distribution
  • Some validators have much lower minimum tips — exploit this

Bundle Simulation

Always simulate before submitting. Jito provides a simulateBundle API that executes your bundle against current state and returns:

  • Success/failure of each transaction
  • Compute units consumed
  • Account state changes (including balance changes)
let simulation_result = jito_client.simulate_bundle(&bundle).await?;

for (i, tx_result) in simulation_result.results.iter().enumerate() {
    match tx_result {
        SimulationResult::Success { cu_consumed, .. } => {
            log::info!("TX {} succeeded, {} CU", i, cu_consumed);
        }
        SimulationResult::Failure { error, .. } => {
            log::warn!("TX {} failed: {}", i, error);
            // Don't submit — save the tip
            return Ok(());
        }
    }
}

Simulation is not free — it adds 10-30ms to your pipeline depending on network conditions. For latency-critical strategies, consider maintaining a local simulation environment using a forked validator state.

Common Failure Modes

Understanding why bundles fail is essential for optimization:

1. Bundle Dropped (No Landing)

The most common failure. Your bundle was received by the block engine but not included in the block. Causes:

  • Tip too low — outbid by competing bundles
  • The slot leader doesn't run Jito — your bundle has no path to inclusion
  • Block was full — total transaction capacity exceeded

Mitigation: Multicast through other submitters simultaneously. If Jito doesn't land it, direct QUIC or another engine might.

2. Transaction Simulation Failure

Your strategy transaction failed during the block engine's simulation. Common causes:

  • Price moved between your detection and the engine's simulation
  • Insufficient funds or wrong account states
  • Compute unit budget too low

Mitigation: Pre-simulate on your end with generous slippage tolerances. Use min_profit checks in your on-chain program rather than exact amounts.

3. Bundle Already Processed

Another searcher's bundle targeting the same opportunity landed first. Your bundle arrives but the state it targets has already changed.

Mitigation: This is pure competition. Win on speed — better data, faster computation, more submitters.

4. Block Engine Timeout

The block engine didn't respond within its processing window. This happens during periods of extreme congestion.

Mitigation: Implement connection pooling with automatic reconnection. Maintain persistent gRPC streams to the block engine rather than creating new connections per submission.

Advanced Patterns

Backrunning Bundles

Include a target transaction (from another user) as the first transaction in your bundle, followed by your strategy transaction. This guarantees your transaction executes immediately after the target:

// Backrun bundle: target TX first, then our arb TX, then tip
let bundle = vec![target_tx, our_arb_tx, tip_tx];

This is the standard pattern for backrun MEV — you detect a large swap, simulate the state after it executes, build your arb transaction against that post-state, and bundle them together.

Multi-Strategy Bundles

You can include up to 5 transactions in a bundle. Use this for multi-step strategies:

  • TX 1: Flash loan borrow
  • TX 2: Swap on DEX A
  • TX 3: Swap on DEX B
  • TX 4: Flash loan repay + profit
  • TX 5: Tip

However, prefer combining steps into a single transaction via a custom on-chain program when possible. Fewer transactions means lower CU consumption and simpler failure handling.

Partial Fill Protection

When your strategy involves multiple swaps, include profit verification in the last strategy transaction (or in each transaction). If the total profit after all swaps is below your minimum threshold, the transaction fails and the bundle reverts — saving your tip.

Connection Management

Production Jito integration requires careful connection management:

  • Persistent gRPC streams: Don't create new connections per bundle. Maintain a persistent connection pool with health checks.
  • Multiple endpoints: Jito operates multiple block engine endpoints across regions. Connect to all and route to the fastest responding one.
  • Automatic reconnection: Implement exponential backoff reconnection for dropped connections.
  • Rate limiting: Jito enforces rate limits per API key. Track your submission rate and backpressure when approaching limits.

Monitoring Jito Performance

Track these metrics for every bundle submission:

  • Submission-to-confirmation latency — How long from submission to on-chain confirmation
  • Landing rate by tip amount — Helps calibrate your tip curve
  • Landing rate by slot leader — Identifies which validators are best for Jito
  • Simulation vs actual result — Detects simulation accuracy drift
  • Failed bundle reasons — Categorize failures for targeted optimization
