How to Build a MEV Bot on Solana

Complete developer guide — from architecture decisions to production deployment. Everything you need to go from zero to profitable extraction.

April 10, 2026 12 min read Developer Guide

Building a MEV bot on Solana is fundamentally different from Ethereum. There's no public mempool to monitor. Transactions are processed in parallel. Block times are 400ms, not 12 seconds. These differences demand a completely different architecture — and that's what this guide covers.

This is not a theoretical overview. Every component described here exists in production systems processing thousands of opportunities daily. We'll walk through the complete architecture, explain the trade-offs at each layer, and provide concrete implementation guidance in Rust.

Prerequisites

Before you start, you need:

  • Rust proficiency — You don't need to be an expert, but you should be comfortable with async Rust, Tokio, and the borrow checker. Python/TS bots exist but cannot compete on latency.
  • Solana fundamentals — Accounts model, transactions, instructions, compute units, and priority fees. The Solana docs cover this well.
  • RPC and Geyser access — You'll need a Geyser gRPC endpoint (Helius, Triton, or your own validator) and Jito block engine access.
  • Infrastructure — A server in the same region as major Solana validators (typically US East or Amsterdam). Bare metal preferred over cloud for consistent latency.

Architecture Overview

A production MEV bot has three distinct layers that operate as a pipeline:

  1. Data Layer — Ingests real-time on-chain state changes
  2. Strategy Layer — Detects profitable opportunities from state changes
  3. Execution Layer — Constructs and submits transactions

Each layer runs as independent async tasks connected by bounded channels. This separation lets you swap strategies without touching the data pipeline, add submitters without changing strategy logic, and scale each layer independently.

Step 1: The Data Pipeline

On Solana, the primary data source for MEV bots is Geyser gRPC (also known as Yellowstone). Unlike RPC polling, Geyser pushes account updates to your bot as they happen — often before the transaction is confirmed in a block.

Setting Up Geyser gRPC

The Yellowstone gRPC plugin streams account updates, transaction notifications, and slot updates. For MEV, you primarily care about account updates on DEX pool accounts.

// Subscribe to all Raydium AMM pool accounts
let subscribe_request = SubscribeRequest {
    accounts: HashMap::from([(
        "raydium_pools".to_string(),
        SubscribeRequestFilterAccounts {
            owner: vec![RAYDIUM_AMM_PROGRAM_ID.to_string()],
            account: vec![],
            ..Default::default()
        },
    )]),
    ..Default::default()
};

The critical optimization: subscribe by program owner, not individual accounts. This gives you updates for every pool under that program, including newly created ones, without maintaining a manual account list.

Account Decoding

Raw account data arrives as byte arrays. You need to decode them into typed structures for each DEX:

  • Raydium AMM V4 — 752-byte account layout with token reserves at known offsets
  • Raydium CLMM — Concentrated liquidity with tick arrays that need separate account loads
  • Orca Whirlpools — Similar CLMM structure with different tick spacing conventions
  • Meteora DLMM — Bin-based liquidity with active bin tracking

Use zero-copy deserialization wherever possible. Anchor's AccountDeserialize works but copies data. For hot-path decoding, read fields directly from byte offsets:

fn decode_raydium_amm_reserves(data: &[u8]) -> (u64, u64) {
    let token_a = u64::from_le_bytes(
        data[RESERVE_A_OFFSET..RESERVE_A_OFFSET + 8]
            .try_into().unwrap()
    );
    let token_b = u64::from_le_bytes(
        data[RESERVE_B_OFFSET..RESERVE_B_OFFSET + 8]
            .try_into().unwrap()
    );
    (token_a, token_b)
}

State Management

Your bot needs an in-memory representation of all monitored pool states. Use a DashMap<Pubkey, PoolState> or similar concurrent map that allows lock-free reads. When a Geyser update arrives, decode the new state and update the map. The strategy layer reads from this map without blocking the data pipeline.

Step 2: Strategy Engine

The strategy engine watches for state changes and identifies profitable opportunities. The most common MEV strategies on Solana:

Cross-DEX Arbitrage

The bread and butter of Solana MEV. When the same token pair has different prices on two DEXes, buy on the cheaper one and sell on the expensive one. The key challenge is speed — these gaps close in milliseconds.

Implementation approach:

  1. Build a token graph where edges are DEX pools
  2. On every pool state update, recalculate quotes for connected edges
  3. Run negative cycle detection (modified Bellman-Ford) to find profitable paths
  4. For each profitable cycle, calculate optimal input amount using binary search

Liquidations

Monitor lending protocols (Marginfi, Kamino, Solend) for accounts approaching liquidation thresholds. When oracle prices push a position below the health factor threshold, submit a liquidation transaction. Profit comes from the liquidation bonus — typically 5-10% of the position value.

Backrunning

Watch for large swaps that move pool prices significantly, then immediately arbitrage the resulting price discrepancy across DEXes. On Solana, this requires Jito bundles to guarantee your transaction lands immediately after the target.

Opportunity Scoring

Not every detected opportunity is worth executing. Your strategy engine should score each opportunity by:

  • Expected profit — after all fees, tips, and slippage
  • Competition density — how many other bots are likely targeting this
  • Execution complexity — number of instructions, CU requirement
  • Historical landing rate — for similar opportunities

Step 3: Transaction Construction

Once a profitable opportunity is identified, you need to build a transaction that captures it. This is where Solana-specific nuances matter most.

Compute Unit Optimization

Every transaction needs a compute unit budget. Too low and it fails. Too high and you waste priority fee spend. The optimal approach:

  1. Simulate the transaction first (via simulateTransaction or local execution)
  2. Take the simulated CU consumption
  3. Add a 10-15% buffer
  4. Set SetComputeUnitLimit to this value

Priority Fee Calculation

Priority fees (via SetComputeUnitPrice) determine your transaction's position in the scheduler queue. Calculate the fee based on:

  • Recent priority fee distribution for your target accounts (Helius priority fee API)
  • Expected profit — never pay more in fees than you'll make
  • Competition level — hot opportunities need higher fees

Atomic Execution with On-Chain Programs

For complex strategies (multi-hop arb, flash loan arb), wrap your logic in a custom Solana program that executes atomically. The program pattern:

// Pseudo-code for atomic arbitrage program
pub fn execute_arb(ctx: Context<ArbAccounts>, min_profit: u64) -> Result<()> {
    // Step 1: Record starting balance
    let balance_before = ctx.accounts.token_account.amount;

    // Step 2: Execute swap A (CPI into DEX A)
    swap_on_dex_a(ctx, amount_in)?;

    // Step 3: Execute swap B (CPI into DEX B)
    swap_on_dex_b(ctx, intermediate_amount)?;

    // Step 4: Verify profit
    let balance_after = ctx.accounts.token_account.amount;
    require!(
        balance_after >= balance_before + min_profit,
        ArbError::InsufficientProfit
    );

    Ok(())
}

The require! at the end is crucial — if the arb isn't profitable after execution (maybe prices moved during the slot), the entire transaction reverts. You only pay the base transaction fee, never losing capital.

Step 4: Submission and Landing

Transaction submission on Solana requires multicasting through multiple channels to maximize landing probability.

Jito Bundles

For most MEV strategies, Jito is the primary submission channel. Bundle your transaction with a tip transaction and submit to the block engine. The tip determines your priority among other bundle submitters. See our complete Jito bundles guide for detailed implementation.

Multicast Strategy

Don't rely on a single submitter. Fire through every available channel simultaneously:

  • Jito block engine (with bundle)
  • Helius staked connections
  • bloXroute BDN
  • Direct QUIC to leader TPU
  • Any additional providers (Astralane, ZeroSlot, Nozomi)

Different submitters have different characteristics — Jito guarantees ordering, QUIC is fastest but unordered, bloXroute has wide propagation. Using all of them gives the best probability of landing.

Step 5: Monitoring and Optimization

A deployed bot is not a finished bot. Continuous monitoring and optimization separate profitable operations from unprofitable ones.

Key Metrics to Track

  • Opportunity detection rate — How many profitable signals per minute/hour
  • Execution rate — What percentage of detected opportunities lead to submitted transactions
  • Landing rate — What percentage of submitted transactions land on-chain
  • Profit per landed transaction — After all fees and tips
  • Latency percentiles — p50, p95, p99 for each pipeline stage

Infrastructure

Use InfluxDB for time-series metrics and Grafana for dashboards. Set up alerts for:

  • Landing rate drops below threshold (might indicate network changes or new competition)
  • Latency spikes (infrastructure issues)
  • PnL going negative (strategy needs adjustment)
  • Submitter failures (API issues with a provider)

Step 6: Deployment

Production deployment considerations:

  • Server location — Co-locate with major validators. US East (Virginia/New York) and Amsterdam are the primary hubs. Bare metal over cloud for consistent latency.
  • Process management — Use systemd with automatic restart. The bot should recover from crashes within seconds.
  • Configuration — All strategy parameters (min profit, max tip, CU buffers) should be hot-configurable without restart. Use a TOML/YAML config file watched for changes.
  • Logging — Structured JSON logs with correlation IDs per opportunity. Essential for post-mortem analysis.
  • Key management — Never hardcode private keys. Use environment variables or a secrets manager. Consider hardware wallets for high-value operations.

Common Mistakes

After building and deploying dozens of MEV systems, these are the mistakes we see most often:

  1. Using TypeScript or Python — The language overhead alone costs 10-50ms. In MEV, that's an eternity. Use Rust.
  2. Polling RPC instead of streaming — getAccountInfo polling adds 50-200ms of latency versus Geyser push updates.
  3. Single submitter — Using only Jito or only QUIC. Multicast through everything available.
  4. No simulation — Submitting transactions without simulating first wastes tips and fees on transactions that will fail.
  5. Static tip amounts — Tips should be dynamic, based on expected profit and recent landing rates.
  6. No circuit breakers — A bug in your strategy can drain your wallet in minutes without automated kill switches.
